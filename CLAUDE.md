# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**sc-sim-tools** is a production-grade analytics pipeline for **FlowForm** — a fictional Polish B2B industrial valve manufacturer.

The pipeline ingests JSON files generated by a separate simulation engine (`sc-sim`, path hardcoded in `.env` as `DATA_DIR`), transforms them through a **medallion architecture** (Bronze → Silver → Gold), and delivers analytics-ready tables consumed by **Power BI dashboards** and **Excel reports**.

The data store is **PostgreSQL**. Orchestration with **Dagster** is the final phase — build the full pipeline first, wire it to Dagster last.

A Python 3.12 virtual environment (`venv/`) is present. Install dependencies into it as needed.

## Coding Standards

- **Write the minimum code required.** No extra error handling, no helper abstractions, no config options, no type annotations beyond what is asked.
- **If something is unclear, ask** before writing code. A short clarifying question is better than a wrong assumption.
- **No speculative generalization.** Do not design for requirements that haven't been stated.
- All SQL lives in `.sql` files or inline strings inside pipeline scripts — no ORM models.
- Python 3.12, `psycopg` (v3) for Postgres.

## Architecture — Medallion Layers

```
DATA_DIR/ (JSON source — sc-sim output, read directly)
  │
  ▼
┌─────────────────────────────────────────────────────┐
│  BRONZE  —  schema: raw                             │
│  Raw JSON → Postgres, no transformation             │
│  raw.events          (all event files)              │
│  raw.ingested_files  (idempotency tracking)         │
│  dim.*               (master reference data)        │
└─────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────┐
│  SILVER  —  schema: stg                             │
│  Typed, deduplicated, normalised, FX-converted      │
│  One table per entity/event type                    │
└─────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────┐
│  GOLD  —  schema: mart                              │
│  Business-level facts, SCD2 dims,                   │
│  pre-aggregated metrics for Power BI / Excel        │
└─────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────┐
│  PHASE 4 — Dagster Orchestration (last)             │
│  Wrap all scripts as @asset, add sensors, schedule  │
└─────────────────────────────────────────────────────┘
```

## Phase 1 — Bronze Layer

**Goal:** land every raw JSON file into Postgres with zero transformation; guarantee idempotency so re-runs are safe.

### Scripts

| Script | Description |
|---|---|
| `src/event_aggregator.py` | Scans `DATA_DIR/{adm,erp,forecast,tms}/YYYY-MM[-DD]/*.json`; tracks processed files in `raw.ingested_files` (SHA-256 hash of file content); deduplicates intra-file events in Python using payload hashes; per-file transactions; does not alter `raw.events` schema |
| `src/load_master.py` | Upserts catalog, customers, carriers, warehouses into `dim.*` via `ON CONFLICT DO UPDATE`; idempotent by design |

### Idempotency rules

- **File-level:** `raw.ingested_files (file_path TEXT PK, file_hash CHAR(64))` — skip if path + hash already recorded. A re-delivered file with a new hash (corrected data) re-ingests and overwrites the file record.
- **Row-level:** Python `seen_hashes` set within each file catches exact payload duplicates before any SQL is issued.
- **Transaction scope:** one transaction per file — either all rows + the file record land or nothing does. A single bad file rolls back and logs; remaining files continue.
- **Schema:** `raw.events` columns are never changed by the aggregator.

### Folder patterns handled

- `YYYY-MM-DD` — ERP and TMS daily folders
- `YYYY-MM` — ADM and Forecast monthly folders (event_date set to first day of month)

## Phase 2 — Silver Layer

**Goal:** parse `raw.events.payload` (JSONB) into typed, named columns; apply all business rules and data-quality fixes; produce one staging table per entity.

Every staging table has:
- `stg_id BIGSERIAL PK`
- `raw_event_id BIGINT REFERENCES raw.events(id)` — lineage back to bronze
- `ingested_at TIMESTAMPTZ DEFAULT now()`
- `_dq_flags TEXT[]` — data-quality issues found (missing fields, unit conversions applied, schema-evolution backfills, etc.)

### Staging tables

| Table | Source event_type | Key transformations |
|---|---|---|
| `stg.orders` | `customer_orders` | Flatten header; one row per order; parse date fields; normalise status enum |
| `stg.order_lines` | `customer_orders` | One row per order × line; cast qty fields to INT; tag backordered lines |
| `stg.exchange_rates` | `exchange_rates` | EUR/PLN and USD/PLN as typed NUMERIC columns; deduplicate by (date, currency_pair) |
| `stg.inventory_snapshots` | `inventory_snapshots` | Flatten positions array; one row per (snapshot_date, warehouse, sku); NUMERIC qty fields |
| `stg.inventory_movements` | `inventory_movements` | Cast direction/reason_code; link to order if reference present |
| `stg.credit_events` | `credit_events` | Cast amounts to NUMERIC; tag hold vs release |
| `stg.production_completions` | `production_completions` | Grade enum validation; link to warehouse |
| `stg.backorder_events` | `backorder_events` | Cast qty; boolean escalated_to_express |
| `stg.payments` | `payments` | Cast amounts; payment_behaviour enum |
| `stg.supply_disruptions` | `supply_disruptions` | Parse severity, expiry date, active flag |
| `stg.master_data_changes` | `master_data_changes` | change_type enum; old/new value as JSONB |
| `stg.production_reclassifications` | `production_reclassifications` | from_grade / to_grade; link to batch |
| `stg.loads` | `loads_08`, `loads_14` | Normalise weight to kg (BalticHaul lbs → kg ÷ 2.20462); deduplicate by load_id across both sync windows using DISTINCT ON (load_id ORDER BY sync_window DESC keeps 14:00 version) |
| `stg.carrier_events` | `carrier_events` | severity enum; rate_delta_pct NUMERIC |
| `stg.returns` | `returns` | One row per RMA; parse lifecycle status; link order + customer + carrier |
| `stg.return_lines` | `returns` | One row per RMA line; qty, reason, resolution |
| `stg.demand_plans` | `demand_plans` | Flatten horizon periods; one row per (plan_month, product_group, segment, period) |
| `stg.inventory_targets` | `inventory_targets` | One row per (month, sku, warehouse) |
| `stg.demand_signals` | `demand_signals` | signal_type, magnitude, confidence, horizon_weeks typed |
| `stg.demand_forecasts` | `demand_forecasts` | One row per (forecast_month, product_group, period); lower/upper bounds |
| `stg.order_cancellations` | `order_cancellations` | Cast qty; cap `quantity_cancelled > 1,000,000` to NULL with `'qty_overflow'` in `_dq_flags`; link to order and line |
| `stg.order_modifications` | `order_modifications` | `modification_type` enum; `old_value`/`new_value` as TEXT; `line_id` nullable |
| `stg.schema_evolution_events` | `schema_evolution_events` | Field name, sim_day threshold, activation date |

### Key silver rules

- **FX conversion:** all PLN-equivalent amounts computed using `stg.exchange_rates` for the event's date. Both PLN and original-currency amounts stored.
- **BalticHaul weight:** `weight_kg = weight_raw / 2.20462` when `carrier = 'BalticHaul'` and weight_unit is absent or `'lbs'`. Flag in `_dq_flags`.
- **Schema evolution backfill:** `sales_channel` defaults to `NULL` before day-60 activation; `incoterms` defaults to `NULL` before day-120. Use `payload->>'field'` with `COALESCE` — never error on missing fields.
- **Snapshot vs movements reconciliation:** `stg.inventory_snapshots` includes `_reconciled BOOLEAN` set by a daily reconciliation job comparing snapshot on_hand against prior-day snapshot + movements delta. Discrepancies go into `stg.inventory_reconciliation_flags`.
- **Incremental:** each stg table processes only rows where `raw_event_id > (SELECT MAX(raw_event_id) FROM stg.<table>)`. Upsert on natural key where entities can change (orders, loads); append-only for immutable events (payments, credit_events).

## Metrics Catalogue

All metrics are computed in the Gold layer. This is the source of truth for definitions.

### Order-to-Cash

| Metric | Definition |
|---|---|
| **Fill Rate (line)** | `SUM(qty_shipped) / SUM(qty_ordered)` per order line |
| **Fill Rate (order)** | % of orders where all lines fully shipped |
| **Backorder Rate** | `SUM(qty_backordered) / SUM(qty_ordered)` |
| **Perfect Order Rate** | % of orders: fully shipped, on time, no returns, no credit hold |
| **Order Cycle Time** | Business days from order_date to first shipment_date |
| **On-Time Delivery Rate** | % of loads where actual delivery ≤ requested_delivery_date |
| **DSO (Days Sales Outstanding)** | `(accounts_receivable_pln / revenue_pln_period) × days_in_period` |
| **Revenue (PLN)** | `SUM(qty_shipped × unit_price_pln)` |
| **Revenue (EUR)** | Revenue in PLN converted at event-date EUR/PLN rate |
| **Gross Revenue by Segment** | Revenue grouped by customer segment |
| **Express Escalation Rate** | `COUNT(backordered lines escalated_to_express) / COUNT(backordered lines)` |

### Inventory

| Metric | Definition |
|---|---|
| **Days of Supply** | `qty_available / avg_daily_demand_last_30d` per SKU per warehouse |
| **Inventory Turnover** | `COGS_pln / avg_inventory_value_pln` annualised |
| **Stockout Rate** | `COUNT(days qty_available = 0) / COUNT(days)` per SKU |
| **Safety Stock Breach Rate** | `COUNT(days qty_available < safety_stock) / COUNT(days)` |
| **Snapshot vs Movements Accuracy** | `ABS(snapshot_on_hand - (prev_snapshot + movements_net)) / snapshot_on_hand` |
| **Allocation Rate** | `qty_allocated / qty_on_hand` |
| **W01 / W02 Inventory Split** | % of total on-hand per warehouse |
| **Slow-Movers** | SKUs with zero shipments in trailing 30d, on_hand > safety_stock |

### Carrier & TMS

| Metric | Definition |
|---|---|
| **Carrier On-Time %** | `COUNT(on_time_flag=true) / COUNT(loads)` per carrier |
| **Transit Time Actual vs SLA** | `AVG(transit_days_actual) - carrier.transit_days_max` |
| **Carrier Rate Volatility** | `STDDEV(rate_delta_pct)` from carrier_events |
| **Carrier Disruption Frequency** | `COUNT(carrier_events)` per carrier per month |
| **Return Rate by Carrier** | `COUNT(rma lines linked to carrier) / COUNT(shipment lines)` |

### Demand & Planning

| Metric | Definition |
|---|---|
| **Forecast MAPE** | `AVG(ABS(forecast_qty - actual_qty) / actual_qty)` per product_group × period |
| **Forecast Bias** | `AVG(forecast_qty - actual_qty)` — positive = over-forecast |
| **Plan vs Actual %** | `(plan_qty - actual_qty) / plan_qty` |
| **Forecast vs Plan Variance** | `forecast_qty - plan_qty` per product_group × period |
| **Signal Conversion Rate** | `COUNT(signals where matching order arrived within horizon_weeks) / COUNT(signals)` |
| **Plan Confidence Band Coverage** | % of actuals falling within (confidence_low, confidence_high) |

### Production

| Metric | Definition |
|---|---|
| **Grade A Rate** | `SUM(qty_completed WHERE grade='A') / SUM(qty_completed)` |
| **Reclassification Rate** | `COUNT(reclassification_events) / COUNT(production_completions)` |
| **Production Output by SKU** | `SUM(qty_completed)` per SKU per period |
| **Supply Disruption Impact** | Shipment volume delta during active disruption vs baseline |

### Credit & Payments

| Metric | Definition |
|---|---|
| **Credit Utilisation** | `exposure_pln / credit_limit_pln` per customer |
| **Credit Hold Frequency** | `COUNT(credit_hold events)` per customer per month |
| **Average Hold Duration** | `AVG(hold_duration_days)` per customer |
| **Payment On-Time Rate** | `COUNT(payment_behaviour='on_time') / COUNT(payments)` |
| **Outstanding Balance** | Latest `balance_after` per customer |
| **Bad Debt Exposure** | Outstanding balance of customers in `credit_hold` status |

### Returns

| Metric | Definition |
|---|---|
| **Return Rate (units)** | `SUM(qty_returned) / SUM(qty_shipped)` |
| **RMA Resolution Time** | `AVG(resolved_date - requested_date)` in business days |
| **Restock Rate** | `COUNT(resolution='restocked') / COUNT(rma_lines)` |
| **Scrap Rate** | `COUNT(resolution='scrapped') / COUNT(rma_lines)` |
| **Return Rate by Reason** | Distribution of return_reason codes |
| **Return Rate by Customer** | Per-customer return rate vs segment average |

## Known Data Engineering Challenges

Each challenge maps to a specific pipeline rule. Issues marked **[CONFIRMED]** were found by auditing the actual source data.

| Challenge | Detail | Pipeline rule |
|---|---|---|
| **Intra-file duplicate event_ids** **[CONFIRMED]** | 119 files contain records with duplicate `event_id` values | Bronze deduplicates on full payload SHA-256 hash, not `event_id`. `event_id` is not a reliable unique key. |
| **inventory_snapshots have no event_id** **[CONFIRMED]** | All 553 snapshot records use `snapshot_id`, not `event_id` | Bronze dedup uses payload hash. Silver uses `(snapshot_date, warehouse_id)` as natural key. |
| **order_cancellations quantity overflow** **[CONFIRMED]** | ~28 records with `quantity_cancelled` > 1,000,000 (simulator integer overflow bug) | Silver caps to `NULL` and adds `'qty_overflow'` to `_dq_flags`. Never errors — passes through bronze as-is. |
| **Null critical fields in orders** **[CONFIRMED]** | ~3 records with null `order_id`, ~2 with null `customer_id` | Silver skips rows where `order_id IS NULL` or `customer_id IS NULL`, logs them to `stg.dq_rejects`. |
| **demand_signals not in date subdirs** **[CONFIRMED]** | 133 individual `SIG-YYYYMMDD-NNNN.json` files live in `adm/demand_signals/` (not a date folder) | Bronze discovery special-cases this path; date parsed from filename via regex. |
| **loads_20.json — third sync window** **[CONFIRMED]** | ~33 days have a `loads_20.json` (20:00 window) not mentioned in original spec | Bronze `*.json` glob already catches it. Silver `stg.loads` deduplicates across all three windows by `load_id` using `DISTINCT ON (load_id) ORDER BY sync_window DESC` (latest wins). |
| **order_cancellations and order_modifications** **[CONFIRMED]** | 62 + 103 files not in original spec | Bronze ingests as-is. Silver produces `stg.order_cancellations` and `stg.order_modifications` tables. |
| **BalticHaul weight in lbs** **[CONFIRMED]** | `carrier_code = 'BALTIC'`, `weight_unit = 'lbs'` (5 confirmed loads) | Silver: `weight_kg = total_weight_reported / 2.20462` when `weight_unit = 'lbs'`. Flag in `_dq_flags`. |
| **Schema evolution — sales_channel / incoterms** **[CONFIRMED]** | `sales_channel` first appears 2026-03-06 (sim day 60). `incoterms` at sim day 120. | `COALESCE(payload->>'sales_channel', NULL)` — never error on missing fields. |
| **Multiple file cadences** | `YYYY-MM-DD` (ERP/TMS daily), `YYYY-MM` (ADM/Forecast monthly), individual files (demand_signals) | Three discovery paths in `event_aggregator.py`. Monthly folders → `event_date = first day of month`. |
| **Late-arriving data** | `event_date` (folder name) may differ from `payload.timestamp` for TMS events | Silver stores both; facts join on `event_date` for aggregation consistency |
| **Snapshots vs movements** | Snapshot on_hand may not equal prev_snapshot + movements_net | `stg.inventory_reconciliation_flags` captures deltas; `mart.fact_inventory_daily` uses snapshot as authoritative |
| **Load → Shipment → Order hierarchy** | `order_ids` and `shipment_ids` are JSON arrays in load records | `fact_shipments` unnests arrays; `mart.bridge_load_order` handles many-to-many |
| **Backorders + partial fulfillment** | Four qty columns (ordered/allocated/shipped/backordered) change independently | `fact_order_lines` is append-only (new row on every status change) |
| **Currency + exchange rates** | Orders in PLN or EUR; payments in PLN | All amounts stored in both original currency and PLN via `stg.exchange_rates` joined on event date |
| **Returns RMA lifecycle** | Multiple event_subtype records per RMA (`return_requested` → `received_at_warehouse` → `resolved`) | `stg.returns` is one row per RMA event_subtype; `fact_returns` is one row per RMA (latest status), `fact_return_lines` per line |
| **Master data changes (SCD2)** | `master_data_changes.json` carries price updates and customer attribute changes | `mart.dim_customer` and `mart.dim_sku` use `effective_from`/`effective_to` |
| **Retroactive corrections** | A corrected re-delivered file has a different SHA-256 hash | Bronze re-ingests (UPSERT on `raw.ingested_files`); silver `ON CONFLICT DO UPDATE` overwrites |
| **Forecast vs demand plan granularity** | Forecasts at SKU-group level; plans at (product_group, segment) | `mart.fact_demand_vs_actuals` joins with COALESCE to align granularities |
| **Carrier events with null duration/rate** | `end_date`, `duration_days`, `rate_delta_pct` all nullable | Silver uses `COALESCE(payload->>'field', NULL)` — treat as open-ended/no-rate-change |

## Phase 4 — Dagster Orchestration (Last)

**Do not start this phase until Phases 1–3 are fully working and verified.**

**Goal:** wrap the completed pipeline scripts as Dagster assets, add a file sensor to trigger incremental runs when new data lands in `DATA_DIR`, and connect the three layers into a dependency graph.

### Project layout

```
sc_sim_tools/
  __init__.py        # Definitions: all assets, sensors, jobs, resources
  resources.py       # PostgresResource (psycopg connection factory)
  sensors.py         # new_source_file_sensor
  assets/
    bronze/
      raw_events.py  # event_aggregator logic as @asset
      raw_master.py  # load_master logic as @asset
    silver/
      orders.py
      loads.py
      inventory.py
      ...            # one file per staging table group
    gold/
      dimensions.py  # dim_date, dim_customer (SCD2), dim_sku (SCD2), ...
      facts.py       # fact_order_lines, fact_shipments, ...
      aggregates.py  # all mart.agg_* tables
  jobs.py            # bronze_job, silver_job, gold_job, full_refresh_job
```

### Sensor behaviour

`new_source_file_sensor` runs every 30 seconds. It compares the newest file mtime in `DATA_DIR` against the watermark in `raw.ingested_files`. If new files exist, it triggers `bronze_job` → on success triggers `silver_job` → on success triggers `gold_job`. Each job processes only the delta since last run.

### Incremental logic

- **Bronze:** skips files already in `raw.ingested_files` by (file_path, file_hash).
- **Silver:** each asset selects `WHERE raw_event_id > (SELECT COALESCE(MAX(raw_event_id), 0) FROM stg.<table>)`.
- **Gold facts:** `INSERT ... ON CONFLICT DO UPDATE` keyed on natural grain. Aggregates are truncated and rebuilt from silver on each gold run.
